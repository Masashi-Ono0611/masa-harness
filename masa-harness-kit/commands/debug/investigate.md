# 目的
報告されたバグやエラーの原因を体系的に調査し、修正方針を提案する。

# 入力検証

`$ARGUMENTS`が空・未定義の場合：
- エラー表示: `🔴 エラー: 調査対象（エラーメッセージ、issue番号、症状の説明）が必要です`
- 使用例:
  - `/debug:investigate "TypeError: Cannot read properties of undefined"`
  - `/debug:investigate 123` (GitHub issue番号)
  - `/debug:investigate ログイン後にリダイレクトされない`
- **即座に実行を停止**

# 調査プロセス

## Step 1: 情報収集

引数の種類に応じて情報を収集する：

**issue番号の場合:**
```shell
gh issue view $ARGUMENTS -c
```

**エラーメッセージ/症状の場合:**
- エラーメッセージからキーワードを抽出
- コードベース内で関連コードを検索

## Step 2: 原因の絞り込み

以下の手法を組み合わせて調査する：

1. **コード検索**: エラーメッセージ、関連する関数名、変数名でコードベースを検索
2. **変更履歴**: `git log` / `git blame` で関連ファイルの最近の変更を確認
3. **依存関係**: 関連するモジュール・パッケージの依存関係を追跡
4. **設定確認**: 環境変数、設定ファイル、インフラ設定の確認

## Step 3: 仮説の構築

調査結果から、最も可能性の高い原因の仮説を立てる：
- 直接原因（何が壊れているか）
- 根本原因（なぜ壊れたか）
- 発生条件（いつ・どの条件で再現するか）

## Step 4: 調査結果の出力

```markdown
# バグ調査レポート

## 症状
（報告された問題の要約）

## 調査結果

### 直接原因
（何が問題を引き起こしているか）

### 根本原因
（なぜその問題が発生したか）

### 関連コード
- `path/to/file:line` - 問題箇所の説明

### 変更履歴
（問題を導入した可能性のあるコミット）

## 修正方針

### 案1（推奨）
- 修正内容: ...
- 影響範囲: ...
- リスク: ...

### 案2（代替）
- 修正内容: ...
- 影響範囲: ...
- リスク: ...

## 再発防止
- [ ] テスト追加案
- [ ] 監視・アラート追加案
```

## Step 5: 修正に着手するかユーザーに確認
調査結果を共有した後、修正の実装に進むかユーザーに確認してください。
